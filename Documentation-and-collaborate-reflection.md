# Documentation & Collaboration Reflection

## Assumptions in the Original README

- A blank or minimal README implicitly assumes its reader already knows the project. It takes for granted that the audience understands the tech stack, knows how to set up their environment, and is already familiar with the team's workflow and folder structure. It treats documentation as an afterthought rather than a communication tool — essentially speaking only to the person who wrote it.

## How the Updated README Improves Onboarding

- The revised README treats a new developer as its primary audience. It opens with a plain-language project overview so anyone ,regardless of technical background can understand what the system does and who it is for. The getting-started section removes ambiguity by listing explicit prerequisites and step-by-step commands, so a new team member can get the app running without having to ask for help. The repository structure section explains *why* folders and files exist, not just what they are called. Placeholders for future sections signal that the documentation is a living artefact and invite contributors to fill in gaps over time rather than letting them go undocumented.

## What I Learned About Communication Through Pull Requests and Issues

- Pull Requests are not just code delivery mechanisms — they are structured conversations. Writing a PR description forced me to articulate *why* a change was being made, not just *what* changed, which made the intent of the work visible to reviewers. Issue templates taught me that removing ambiguity upfront saves significant back-and-forth later; when a reporter fills in steps to reproduce, expected behaviour, and environment details, another developer can act on the issue immediately. Both practices shift collaboration from implicit and assumed to explicit and documented — which is what makes them valuable in professional teams.
